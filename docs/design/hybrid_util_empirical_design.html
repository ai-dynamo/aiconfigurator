Hybrid Modeling Enhancement: SOL-Utilization Empirical

Branch enhance_hybrid · Module sdk/operations/util_empirical.py · Methodology & validation summary

Replace every op's hard-coded empirical scale_factor with a SOL-utilization (util) read from real collected data. In one line: model in the util = SOL/measured space, not in latency space — util is bounded, smooth, and transferable across structural variants.

1 · Problem & Motivation

The original empirical estimate was a fixed per-op constant:

empirical_latency = SOL(shape) / scale_factor   # scale_factor ∈ {0.4 … 0.8}, hard-coded

That constant is really an "assumed utilization" and is badly off across shapes. Five observations motivated the change:

Obs 1Interpolation needs a full cartesian grid; a missing point errors out — hybrid should fill it.
Obs 2A fixed scaling factor can't be accurate; how do we lightly reuse existing data?
Obs 3SOL% is smoother than latency; good as an extra dimension; extrapolation should clamp to boundary SOL%.
Obs 4New model with missing ops → afraid to use hybrid; a new op could borrow a similar op's sol%.
Obs 5attn / moe minor variants (different head / expert counts) are structurally alike; under the same input they should reuse a similar op's sol%.

2 · Core Method: Model in util Space

util = SOL / measured ∈ (0, 1]  (measured ≥ SOL, guaranteed by _correct_sol)
latency(query) = SOL(query) / util(query)

Why util, not latency

  • Bounded: util∈(0,1]; extrapolation can't go negative or explode.
  • Smooth: util is "kernel efficiency", far lower curvature than latency (which spans orders of magnitude with M·N·K).
  • Transferable: analytic SOL absorbs structural differences (heads / experts / hidden); util keeps only kernel efficiency, ~invariant across structure.

SOL = analytic prior, util = residual

Use analytic SOL as a basis capturing the bulk variation; interpolate only the smooth efficiency residual. The query's SOL is computed exactly, adding a physical prior that raw latency interpolation lacks.

SOL handles quant / prefix / window kinks analytically, so util is even flatter across those boundaries.

3 · Estimation Priority Chain (core methodology diagram)

How a query resolves to a latency. EMPIRICAL mode (and the HYBRID silicon-miss fallback) walks the three-level chain on the right; each level is more accurate than the next:

query(shape)  DatabaseMode?
▼
SILICON — strict interpolation on the collected grid; raise on a missing bracket (→ handed to HYBRID fallback)
▼ EMPIRICAL / HYBRID-miss
① own-slice util — from this op's own collected data; nearest-neighbour / boundary clamp (Obs 2, 3)
latency = SOL(query) / util_own
▼ no data in own slice
② cross-shape transfer — borrow the nearest sibling slice's util, reconstruct with the query's own SOL (Obs 4, 5)
latency = SOL(query) / util_ref(nearest sibling)
▼ no sibling to borrow
③ constant fallback — revert to the op's original scale_factor (keeps zero-data ops unchanged)
latency = SOL(query) / scale_factor

Each level beats the next and keeps a source tag for attribution. The own-data path is identical to pre-change — transfer/constant fire only when data is missing.

4 · Module Structure util_empirical

Extension seam: the sample source is swappable (own / sibling slice / cross-op reference); UtilGrid and estimate stay unchanged — that's what makes "transfer" pluggable.

5 · Cross-Shape Transfer (Obs 5)

"SOL-ratio transfer": the analytic SOL carries the structural difference; util only borrows the shared kernel efficiency.

latency_query(c) = SOL_query(c) / util_ref(c)
  = SOL_query(c) · measured_ref(c) / SOL_ref(c)  # SOL_ref MUST use the reference's own shape

The similarity key was found by POC to be (topk, num_experts) + GEMM size regime, not model lineage (DeepSeek-V3→MiniMax 6.9% beats same-family Qwen30B→Qwen235B 27%).

6 · Validation Methodology (two layers)

① Model-level run_static gate

Run the same model over (bs, seq, pastkv) in SILICON vs EMPIRICAL; treat silicon as reference and report MAPE/p95/max. Measures empirical's model-level accuracy.

② Per-op invariant

At a collected point, SOL/util = SOL/(SOL/measured) = measured, so emp/sil must = 1.0000. Used to confirm wiring when the model-level gate is blocked by data gaps.

Recurring theme: run_static gates are often blocked by collection data gaps (GLM-5 prefix>0, trtllm alltoall table, DSV4 fp8 fmha, attention prefix>0) — a data problem, not a wiring problem; fall back to the per-op invariant.
