Estimating latency for a model you have no data for

Why we model in utilisation space, how perf transfers across shapes and ops, and what happened when we onboarded MiniMax-M3 with zero MSA benchmarks.

aiconfigurator predicts the latency of an LLM deployment — every GEMM, attention, MoE and communication op — from a database of microbenchmarks. The problem: the database is always incomplete. A new model arrives, a new quantisation, a new GPU, a brand-new attention variant — and there is simply no measurement to interpolate. This is the story of how we stopped guessing a fixed fudge factor and started transferring utilisation instead, and how that let us model MiniMax-M3 end-to-end before a single MSA kernel was ever benchmarked.

1 · The hole in interpolation

The accurate path is SILICON mode: look up the measured latency for the exact (op, shape, quant) and interpolate between bracketing points. When the bracket exists, it's great. When it doesn't — a sequence length past the grid, a batch size never collected, an op that simply isn't in the data — interpolation has nothing to stand on.

The old fallback (HYBRID mode) was a fixed scaling factor: take the roofline lower bound (SOL — "speed of light", the time if the GPU ran at peak FLOPs/bandwidth) and multiply by a constant like 1 / 0.8. One number for every shape, every op. It is wrong almost everywhere: a tiny GEMM runs at 15% of peak, a large one at 90%, and a single constant splits the difference by being bad at both.

2 · Model in utilisation space, not latency space

The key move is a change of variable. Instead of modeling latency directly, model the achieved utilisation:

util = SOL / measured_latency # in (0, 1], how close to peak the kernel ran latency = SOL / util # invert to predict

Why this is the right space: util is bounded and smooth. It lives in (0,1], it moves gently and monotonically with problem size (small → low util → climbing → saturating), and it factors out the part we already know exactly (the SOL roofline, which we can compute analytically for any shape). Latency spans four orders of magnitude across shapes; util spans maybe 5×. Interpolating — or transferring — a quantity that varies 5× is far safer than one that varies 10,000×.

Same kernel, two coordinate systems. We fit, interpolate, and transfer the purple curve, then multiply the analytic SOL back in to recover latency.

3 · A ladder of fallbacks

Once perf is expressed as util, "I have no data" stops being binary. There is a graceful ladder, each rung used only when the one above has nothing:

  1. Own-data util curve. The op has measurements for this quant — fit util vs size and read it off. (This is just SILICON, re-expressed.)
  2. Cross-shape transfer. No data at this shape, but the op was measured at other shapes. Find the nearest neighbour in normalised log-space and borrow its util. Similarity is keyed on what actually drives the kernel — e.g. MoE on (topk, num_experts) and the size regime — not on model lineage.
  3. Sibling-version fallback. No data for this op at this backend version? Pull the same op from the nearest sibling version (manifest-gated, newest-first).
  4. Cross-op transfer. A brand-new op with no data at all borrows the util curve of a structurally similar op. util is not directly comparable across op classes (each op's SOL covers a different fraction of the real work), so this rung carries a manual per-op-pair scale k: latency = SOL / (util_ref · k). Default k = 1, set deliberately when you know the two ops' efficiency offset.
  5. Constant fallback. The old fixed factor, now the last resort instead of the first.
The constant didn't disappear — it got demoted. Every rung above it is data-grounded; you only hit the constant when literally nothing structurally related was ever measured.

4 · Does it actually track silicon?

We validated rung-by-rung with run_static, comparing the empirical path against true silicon on models that do have full data (so we can score the error):

The headline is rung 2: when we deliberately deleted the MoE rows for a shape and forced a transfer, the util-borrowing path landed at 15% error where the old constant was off by 113%. Same input, an 8× improvement, purely from transferring a bounded quantity instead of inventing one.

5 · The real test: MiniMax-M3 with zero MSA data

Then a model showed up that exercises the whole ladder at once. MiniMax-M3 introduces MSA (MiniMax Sparse Attention): a cheap per-block indexer scores KV blocks, the top-k blocks (16 × 128 = 2048 tokens) are selected, and full GQA attention runs over only those. We had no MSA benchmarks at all.

But MSA is structurally a GQA cousin of DSA (DeepSeek sparse attention), which we do have data for. So we:

  • Derived the MSA SOL by hand — three groups: the projection GEMMs, the per-block fp8 indexer, and the sparse attention itself (pair count saturating at the top-k cap). Prefill is compute-bound; decode is memory-bound and, crucially, context-independent because the KV touched is capped at the top-k selection.
  • Transferred util from DSA (rung 4, k = 1) — MSA borrows DSA's measured efficiency curve, scaled by the analytic SOL ratio.

Result, pure HYBRID, no tuning: M3 decode TPOT 23.5 ms vs the published sglang B200 number of 24.1 ms — within 2%. And the flat-in-context decode shape predicted by the SOL was confirmed independently by GLM-5 silicon (sparse decode really is context-independent).

6 · An honesty section: the data you think you have

The interesting failure came when we compared M3 against its dense sibling M2.7 under NVFP4 and the M3 decode line looked suspiciously, perfectly flat. It was the constant fallback in disguise: under NVFP4 the MSA transfer had nothing to grab, so it silently bottomed out at rung 5.

Chasing it down was a lesson in reading your own database. The DSA module's mla_dtype column is always bfloat16 — which made it look like there's no quantised DSA data anywhere. But the quant actually lives in a separate gemm_type column (bfloat16 / fp8_block / nvfp4), and it's keyed per architecture. The NVFP4 DSA data exists richly — just not for the exact (architecture, backend-version) cell M3's transfer was pointing at on that one SKU. fp8_block, on the other hand, had real data on the same cell, so it transfers honestly.

"No data" and "data under a column I wasn't looking at" produce identical-looking flat lines. A graceful fallback ladder is only trustworthy if it's loud about which rung it landed on.

So the faithful M2.7-vs-M3 comparison uses fp8_block, and it tells a clean story:

M2.7's full attention grows with context (14.5 → 42 ms); M3's MSA is capped by the top-k selection and stays flat (~18–24 ms). Short context favours the smaller M2.7; beyond ~6k the sparse attention wins — at 16k, M3 is 2.3× faster per token. The entire M3 curve comes from a derived SOL plus a DSA-transferred util — no MSA measurements.

7 · Takeaways

  • Pick a bounded coordinate. Modeling util (∈ (0,1]) instead of latency (∈ 10,000×) makes interpolation and transfer behave.
  • Make "no data" a ladder, not a cliff. Own-shape → cross-shape → sibling-version → cross-op → constant. Each rung is data-grounded until the last.
  • Transfer on what drives the kernel. MoE similarity is (topk, experts), not the model name. MSA borrows DSA because they share a sparse-attention structure, not a vendor.
  • Be loud about the rung. A silent constant fallback is indistinguishable from a real estimate until it bites — instrument which rung fired.
  • Derive the SOL when the kernel is new. A hand-derived roofline plus a transferred util got MiniMax-M3 within 2% of measured, before MSA was ever benchmarked.
