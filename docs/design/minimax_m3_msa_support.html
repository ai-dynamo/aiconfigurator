MiniMax-M3 Support — MSA op + cross-op transfer

Pure HYBRID · new MSA operator (SOL derived) · transfers from DSA · validated vs sglang B200

Goal: model MiniMax-M3 (text decoder) end-to-end with no MSA perf data. M3 introduces MSA (MiniMax Sparse Attention). We derive its SOL, run it in pure HYBRID, and let it transfer from DSA's measured utilisation via the util_scale hook. Result: TPOT within 2% of the real sglang B200 number, with no tuning.

1 · What MSA is (and why it transfers to DSA)

MSA (github.com/MiniMax-AI/MSA) is structurally a GQA version of DSA: a cheap per-block "dense proxy" indexer scores KV blocks, the top-k blocks are selected, and full attention runs over only the selected tokens. The only real difference from DSA is the main attention — GQA (M3: 64 q / 4 kv heads, head_dim 128, partial rope) instead of DSA's MLA-compressed attention. The sparse selection (index_topk = 16 blocks × 128 = 2048 tokens) is the same.

DSA = same pipeline but MLA attention + 32–64 index heads. Because both are sparse-top-k families with similar kernel efficiency, MSA can borrow DSA's utilisation almost 1:1.

2 · MSA SOL (derived from GQA / DSA / DSV4 logic)

No MSA silicon data exists, so SOL must be analytic. It only needs to capture the shape trend (how latency scales with b, s, prefix); the absolute level is set by the transfer. Three groups, same split as DSA/DSV4:

tokens = b·s full_s = prefix + s (context) GEMM group = 2·tokens·hidden·(num_heads·d) # Q + 2·tokens·hidden·(2·num_kv_heads·d) # K,V (GQA → by num_kv_heads) + 2·tokens·(num_heads·v_d)·hidden # O + 2·tokens·hidden·(idx_heads·idx_d) # indexer Q indexer (fp8)= 2·tokens·idx_heads·idx_d·num_blocks # per-BLOCK scoring (÷ block_size) sparse attn = 2·num_heads·(qk_d + v_d)·pairs # QK^T + AV pairs = top-k saturated causal count (DSA's formula) KV cache mem = b·num_kv_heads·min(full_s,topk)·(qk_d+v_d)·kv_bytes # GQA cache by kv heads SOL = max( Σ group_ops / tc_flops , total_mem / mem_bw )

Key choices: KV cache by num_kv_heads (GQA), compute by num_heads; indexer scores per block (MSA's proxy pass), ~block_size× cheaper than DSA's per-token; sparse pair count reuses DSA's causal-ramp-then-saturate-at-topk count.

3 · Empirical = cross-op transfer from DSA

latency_MSA(b,s) = SOL_MSA(b,s) / ( util_DSA(b,s) · k ) util_DSA = DSA SOL / DSA silicon (at the same workload) k = msa_dsa_scale_k (util_scale hook, default 1.0, user-tunable)

SOL carries the structural difference (GQA vs MLA, 4 vs 64 index heads); DSA's measured util carries the shared sparse-attention kernel efficiency. Falls back to a constant when DSA data is absent. This reuses the util_scale seam added for cross-op transfer — k pulls the level, SOL the shape.

4 · Integration (text decoder only)

New op

operations/msa.py: ContextMSAModule / GenerationMSAModule. SILICON raises (no data); HYBRID/EMPIRICAL → DSA transfer.

New model

models/minimax_m3.py MiniMaxM3Model (family MINIMAXM3), mirrors DeepSeekV32Model with MSA. Registered MiniMaxM3SparseForConditionalGeneration (multimodal → text_config unwrap) + MiniMaxM3ForCausalLM.

M3 = hidden 6144, 60 layers, 64 q / 4 kv heads (d 128), 128 experts / topk 4 / 1 shared. Vision tower skipped (text decoder only); first-3-dense approximated as all-MoE (matches GLM-5/DSV3.2).

5 · Validation vs sglang B200 cookbook

sglang cookbook (MiniMax-M3, B200, mxfp8, TP8, single node, ISL 2048 / OSL 256, concurrency 64): TPOT 24.1 ms, TTFT 1580 ms, 265 tok/s/GPU. Our M3 in pure HYBRID (MoE from real silicon, MSA transferred from DSA at k=1.0), fp8_block ≈ mxfp8, bs 64, kv ≈ 2176:

metricM3 hybrid (est)sglang B200ratio
TPOT (decode step)23.5 ms24.1 ms0.98

within 2% — no k tuning

Decode breakdown: MoE 12.4 ms (real silicon data) + MSA 10.6 ms (DSA transfer) + norms. k≈1 works because MSA↔DSA are both sparse-top-k families with near-equal utilisation (cf. MHA↔MLA which needed k≈1.41). TTFT not compared directly — sglang's 1580 ms is concurrency-64 batched prefill; single-prompt context ≈ 47 ms, which needs serving-level (continuous-batching) modeling.

6 · MiniMax-M2.7 vs M3 on the same B200 (fp8_block)

A direct apples-to-apples on b200/trtllm, tp8, fp8_block: M2.7 in SILICON (MOE + full GQA attention) vs M3 in pure HYBRID (MoE + MSA, MSA transferred from DSA). fp8_block is used because b200 has no NVFP4 DSA data — under NVFP4 the MSA transfer would silently fall back to a constant; fp8_block has DSA data so the transfer is real (and M2.7 has fp8_block silicon too). M3 is the larger model (hidden 6144 vs 3072, 64 vs 48 heads) so prefill and short-context decode are heavier, but its MSA caps decode KV at the top-2048 selection, so TPOT stays flat as context grows while M2.7's full attention grows linearly — they cross around ~6k.

M2.7's full attention grows with kv (14.5 → 42 ms); M3's MSA is context-independent (top-k cap, ~18–24 ms). Short context favours M2.7 (M3 is the bigger model); beyond ~6k the sparse MSA wins — at kv 16384 M3 is 2.3× faster per token. The hybrid M3 reproduces this purely from the MSA SOL (pairs capped at index_topk) + the real DSA-transferred util; no MSA measurements. Figure: poc/hybrid_util/compare_m27_m3.png.

6.1 · Decode sanity check vs real DSA (GLM-5, SILICON)

Is M3's flat decode real, or an artifact? Cross-check against a model that has sparse-attention silicon data. GLM-5 (real DSA) in SILICON also shows non-growing decode — sparse attention is genuinely context-independent (top-k cap). The faithful M3 decode (fp8_block, where DSA data exists so the transfer is real) sits below GLM-5's (lighter MSA: 4 vs 32 index heads, GQA vs MLA). The NVFP4 M3 column is the constant fallback (no DSA NVFP4) and should not be read as a transfer.

kv 409651.2 ms21.2 ms10.3 ms
kv 819240.7 ms17.5 ms10.3 ms
kv 1638449.9 ms17.9 ms10.3 ms

All three are flat in kv → the sparse-decode behaviour is physically real, confirmed by real DSA silicon. Take-away: cross-op transfer needs the reference op (DSA) to have data at the query's quant — otherwise it degrades to the constant. So either collect NVFP4 DSA data, or set msa_dsa_scale_k against the fp8_block transfer.
